设计模式-策略模式

|字数总计:1.4k|阅读时长:5分钟|阅读量:|评论数:

摘要：

  1. 策略模式
  2. 单一职责原则
  3. 开放-封闭原则
  4. 依赖倒转原则
  5. 里氏代换原则

一、策略模式

1、问题引出

需设计一个水果店的收费系统，再计算出总价格后可以正常收费、打折或者满减；

简单工厂模式实现：

前文所述：简单工厂模式是解决在创建对象时不知道创建哪一个的问题，而这个问题的核心更在于如何灵活的变换要调用的方法，而不是创建哪一个对象。

2、策略模式引出

策略模式（Strategy），定义了一系列的算法家族，分别封装起来，它们之间可以互相替换；算法的变化不会影响到使用算法的客户

3、解决问题

二、单一职责原则

生活的例子；早些年手机功能不是这么强的时候，出门需要带mp3听歌，mp4看视频，钱包付款，相机拍照，等等。而现在出门只需要带手机就可以完成这些所有的功能；手机的设计违反单一职责原则，比如在大街上看到一个好看的小姐姐，用相机可以很快的拍下来，而使用手机在打开相机的时候（或者你正在打游戏）她就可能已经走远了（勉勉强强的例子）；

单一职责原则（SRP）：就一个类而言，应该仅有一个引起它变换的原因。

Single responsibility principle：There should never be more than one reason for a class to change

三、开放-封闭原则

开发过程难免会有需求的变化，不说是为客户开发，自己开发也是经常变动需求，而设计出易于拓展的软件显得尤为重要，不至于在修改需求时将整个软件系统推翻重做；

开放-封闭原则（OCP）：软件实体（类、函数、模块等等）应该可以拓展，但是不可修改

Open-Closed Principle ：Software entities(classes,modules,functions etc) should open for extension ,but close for modification.

四、依赖倒转原则

依赖倒转原则（DIP）：

A、高层依赖不应该依赖底层模块，两个都应该依赖抽象

B、抽象不应该依赖细节，细节应该依赖抽象（说白就是针对接口编程而不是针对实现）

Dependency inversion principle

A、High level modules should not depend upon low level modules. Both should depend upon abstractions. 1

B、 Abstractions should not depend upon details. Details should depend upon abstractions。

开发中经常把一些常用的方法封装起来，如访问数据库的代码，由高层模块调用，这就是高层模块依赖底层模块；在开发新的项目时直接调用这些代码；

但如果客户要求更改需求，如数据库或存储信息方式，这时当我们要使用高层模块时，高层模块和底层模块是绑定在一起就无法使用，就麻烦了；就像cpu，内存等如果都高度依赖主板，主板一坏直接jj，但事实确实主板这些电脑上的设备都是依赖接口，也就导致电脑维修的时候哪里坏了换哪里就行；

实现：如开发中service dao层都是依赖接口，而不是实现类

五、里氏代换原则

里氏代换原则（LSP）：子类型必须能够替换掉他们的父类型

Liskov Substitution Principle：Inheritance should ensure that any property proved about supertype objects also holds for subtype objects

子类可以扩展父类的功能，但不能改变父类原有的功能；里氏代换原则解决了什么时候应该使用继承，什么时候不应该使用继承的问题；如果设计的类关系违反了里氏代换原则，就应该重新设计它们之间的关系；只有子类可以替换父类而软件系统不受影响的时候，父类才能真正实现复用，子类也能在父类的基础上拓展功能
